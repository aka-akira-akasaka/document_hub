# 開発本部（DevHQ）組織定義

> document_hub内の開発組織。Claude Codeの全能力を活用し、SaaS開発・社内ツール開発を推進する。

---

## ミッション

銀行渉外・SaaS推進省の活動を技術面から支援し、以下を実現する：

1. **銀行向けSaaSプロダクト**の設計・実装・運用
2. **社内業務自動化ツール**の開発による生産性向上
3. **PoC・プロトタイプ**の高速開発による仮説検証の加速

---

## 開発ワークフロー

```
要件定義 → 設計（Plan Mode） → 承認 → 実装 → テスト → レビュー → デプロイ
```

### 1. 要件定義

- 赤坂さんまたはMinistry各局からの要件をヒアリング
- ゴール・スコープ・制約を明確化
- 受け入れ基準を定義

### 2. 設計（Plan Mode）

- アーキテクチャ設計・技術選定を行う
- 2〜3のアプローチを比較検討（Demand Elegance）
- 設計書を赤坂さんに提示し承認を得る

### 3. 実装

- `projects/<プロジェクト名>/` 配下で開発
- サブエージェントを活用し、メインコンテキストを保護
- コミットは機能単位で小さく行う

### 4. テスト

- ユニットテスト・結合テストを作成
- エッジケースとエラーハンドリングを検証
- セキュリティチェック（入力バリデーション、認証・認可）

### 5. レビュー・デプロイ

- 実装完了後、赤坂さんに動作確認を依頼
- フィードバックを反映
- デプロイ手順を文書化し実行

## 開発標準

### コーディング規約

- **言語選択**: プロジェクト要件に応じて最適な言語を選択（TypeScript推奨）
- **命名規則**: 変数・関数はcamelCase、クラスはPascalCase、定数はSCREAMING_SNAKE_CASE
- **コメント**: コードの「なぜ」を説明。自明な処理にはコメント不要
- **エラーハンドリング**: 外部入力は必ずバリデーション。内部コードはフレームワークの保証を信頼

### アーキテクチャ指針

- **シンプルさ優先**: 抽象化は必要になってから。早すぎる最適化を避ける
- **関心の分離**: ビジネスロジック・データアクセス・UIを分離
- **設定の外部化**: 環境変数や設定ファイルで管理。ハードコーディングしない
- **セキュリティ**: OWASP Top 10を意識。シークレットはコードに含めない

### ディレクトリ規約

```
projects/
├── <プロジェクト名>/
│   ├── README.md          # プロジェクト概要・セットアップ手順
│   ├── src/               # ソースコード
│   ├── tests/             # テストコード
│   ├── docs/              # プロジェクト固有ドキュメント
│   ├── .env.example       # 環境変数テンプレート（秘匿値は含めない）
│   └── package.json等     # 依存管理ファイル
```

## 権限と裁量

### 自律実行可能な業務

- コードの設計・実装・テスト（承認済み設計に基づく）
- ローカル環境での動作確認
- テストの作成・実行
- ドキュメントの作成・更新
- 依存パッケージのインストール（セキュリティ上問題のないもの）
- gitでのコミット・ブランチ操作（ローカル）

### 赤坂さんの承認が必要な業務

- 新規プロジェクトの立ち上げ（スコープ・技術選定の承認）
- 外部サービスとの連携（API キー取得、サードパーティ利用）
- デプロイ・公開
- 大規模なアーキテクチャ変更
- セキュリティに関わる設定変更
