## ミッション

Claude Codeの全能力を活用し、ユーザーの業務に資するプロダクト（社内ツールを含む）を開発する。

## 組織構成

```
開発本部（DevHQ）
├── プロダクト開発課 — SaaS製品の設計・実装・運用
├── ツール開発課     — 社内業務自動化・効率化ツール
└── 品質管理課       — テスト・コードレビュー・CI/CD・セキュリティ
```

詳細な組織定義・ワークフロー・権限: `dev-org/charter.md`

## 開発原則

1. **ユーザビリティ最優先**: 全ての設計・実装判断において、ユーザー体験を最優先する
   - **レスポンス速度**: 3秒以上かかる処理には必ずプログレスバー等の進捗表示を実装し、"処理している感"を提供する
   - **目的直結UI**: ユーザーがその画面で"やりたいこと"を常に想定し、最短動線で到達できるUIを提供する
   - **自明な操作性**: "どこを触ればできるか"が一目でわかる設計とし、説明書を不要とする
2. **設計先行**: 実装前にPlan Modeで設計を承認してもらう
3. **小さく作って早く見せる**: MVP → フィードバック → 改善のサイクル
4. **テスト必須**: 主要ロジックにはテストを書く。リリース前にCIを通す
5. **セキュリティ最優先**: OWASP Top 10を意識し、脆弱性を作り込まない
6. **ドキュメント同梱**: READMEとインラインコメントで自己説明的なコードを書く
7. **日本語統一**: コミットメッセージ・コードコメントはすべて日本語で書く

## プロジェクト管理

- コードプロジェクトは `projects/<プロジェクト名>/` に格納
- 各プロジェクトは独立したディレクトリ構成を持つ
- ブランチ戦略: `main` → `feature/<機能名>` → PR → マージ

---

## Ministry（4局体制）との連携

DevHQはMinistryの4局とは独立した組織だが、以下の点で連携する：

| 連携元 | DevHQへの依頼例 |
|--------|----------------|
| 渉外・アライアンス局 | 銀行向けデモアプリ、PoC開発、API連携 |
| 需要創出・マーケティング局 | LP制作、提案用デモサイト、データ可視化ツール |
| インテリジェンス・監査局 | データ収集スクリプト、分析ダッシュボード |
| 官房・オペレーション局 | タスク自動化ツール、レポート生成スクリプト |
