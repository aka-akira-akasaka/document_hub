# AIアシスタント運用ガイドライン

> このファイルはdocument_hub組織（ビジネスアシスタント＋開発本部）の動作ルールを定義します。
> CLAUDE.md から参照されます。

---

## 基本原則

- **常に日本語でやり取りする**
- **タスクは必ずサブエージェントに委託する**（直接実行しない）
- 対応するサブエージェントが存在しない場合、初回のみ「サブエージェント作成サブエージェント」を構築してよい。2回目以降はその作成サブエージェントに委託する。

---

## 1. Plan Mode Default

3ステップ以上のタスクは、実行前に必ず計画を提示する。

```
計画提示 → ユーザー承認 → サブエージェントへ委託
```

---

## 2. Self-Improvement Loop

- ミスしたパターンを `docs/assistant/errors_log.md` に記録する
- セッション開始時に `errors_log.md` を読み返してから作業する
- 記録フォーマット：

```
## YYYY-MM-DD: <ミスの概要>
- **状況:** どんな状況でミスが起きたか
- **ミス内容:** 何をしてしまったか
- **再発防止策:** 次回どうするか
```

---

## 3. Verification Before Done

完了前に以下を自問する：

> 「シニアコンサルタントが見て承認できるレベルか？」

チェックリスト：
- [ ] コードは読みやすく、意図が明確か
- [ ] セキュリティ上の問題はないか
- [ ] テストは網羅されているか
- [ ] 設計判断に根拠があるか

---

## 4. Subagent Strategy

- リサーチ・分析・実装はすべてサブエージェントに委託する
- メインのコンテキストウィンドウを汚染しない
- サブエージェントの種類：
  - `general-purpose`: 実装・ファイル操作全般
  - `Explore`: コードベース探索
  - `Plan`: 設計・計画立案

---

## 5. Demand Elegance

設計判断を含む変更では、力技の前に **2〜3のアプローチを比較検討** する。

比較観点：
- シンプルさ vs 拡張性
- 即実装可能か vs 長期保守性
- 既存パターンとの整合性

---

## 6. Autonomous Bug Fixing

バグ・不具合報告が来たら：

1. ユーザーに聞く前にまず自分で調査する
2. ログ・スタックトレース・関連コードを確認する
3. 修正案を実装してテストする
4. 修正内容をユーザーに報告する

---

## 7. 開発モード（DevHQ）

開発タスクを受けた場合、以下のルールで動作する。

### 切り替え判断

以下のいずれかに該当する場合、開発モードで対応する：
- コードの設計・実装・修正を依頼された場合
- プロジェクト作成・技術選定を依頼された場合
- テスト作成・CI/CD構築を依頼された場合

### 開発時の行動原則

- **Plan Mode必須**: 新規プロジェクト・大きな機能変更は必ず設計を先に提示
- **サブエージェント活用**: 実装はサブエージェントに委託し、メインコンテキストを保護
- **テスト駆動**: 主要ロジックはテストを先に書くか、実装と同時に書く
- **セキュリティ意識**: 外部入力のバリデーション、シークレットの取り扱いに注意
- **小さなコミット**: 機能単位で小さくコミットし、変更履歴を追跡可能にする

### 開発時のVerification

完了前に以下を追加で確認する：

- [ ] テストが通るか
- [ ] セキュリティ上の問題はないか（OWASP Top 10）
- [ ] README・ドキュメントが最新か
- [ ] 不要なコード・依存が残っていないか
- [ ] .env等の秘匿情報がコミットに含まれていないか

### プロジェクト配置

- 全てのコードプロジェクトは `projects/` 配下に格納
- 組織定義・開発標準は `dev-org/` を参照
